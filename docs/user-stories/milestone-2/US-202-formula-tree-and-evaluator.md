# US-202 — Formula tree and evaluator

**Epic**: E3 Program model & interpreter | **Estimate**: ~5 h | **Depends on**: US-201

**Status**: In Progress

**Story**: As a brick, I want to interpret a formula of literals, arithmetic operators, and variables to a number, mirroring Catroid's evaluator semantics.

## Acceptance criteria
- [ ] `Formula` is an `indirect enum`: `number(Double)`, `variable(String)`, `binary(BinaryOperator, Formula, Formula)`, `unaryMinus(Formula)` — mirroring Catroid's `FormulaElement` tree restricted to the M2 subset (NUMBER, USER_VARIABLE, OPERATOR). Operators: `plus`, `minus`, `mult`, `divide`, `pow`. Unary minus mirrors Catroid's MINUS-with-only-a-right-child. **No `mod` operator**: Catroid declares `Operators.MOD` but its operator interpretation has no case for it (it falls through to the default-zero branch) — the working modulo is `Functions.MOD`, a *function*, and functions are outside the M2 subset. Brackets are omitted from the tree (a parse/round-trip concern for the `.catrobat` importer, stretch scope — precedence is encoded structurally in M2).
- [ ] `interpretDouble(scope:) throws`: recursive evaluation; **throws on NaN only** (Catroid `assertNotNaN` checks `Double.isNaN()` and nothing else — `InterpretationException`). **Division by a zero divisor yields NaN and therefore throws** — Catroid's DIVIDE case explicitly returns NaN whenever the divisor is 0; it never produces ∞ there. NaN is sticky: a NaN operand makes every M2 operator return NaN, so it always reaches the root and throws. **±Infinity never survives as a value**: Catroid post-processes *every* node's result — including operator operands, which are interpreted via `interpretRecursive` before entering the parent — through `normalizeDegeneratedDoubleValues` (`FormulaElement.java:391-394`; `FormulaElementOperations.kt:97-101`, operand path `:325-332`), mapping `+∞ → Double.MAX_VALUE`, `−∞ → −Double.MAX_VALUE`, NaN unchanged. So a `pow` overflow (e.g. `1e308^2`) observably evaluates to `Double.greatestFiniteMagnitude` — not `∞` — and `pow(1e308, 2) + 1` evaluates to ≈`Double.greatestFiniteMagnitude` (the `+1` is far below the ulp), not 1. *(Amended 2026-07-19: the original criterion claimed ∞ propagates and that a non-finite operand entering PLUS/MINUS/MULT/DIVIDE is coerced to 0 via Catroid's `BigDecimal.valueOf` catch — source verification showed that catch is unreachable dead code: ∞ is normalized away before any operand conversion, and NaN short-circuits earlier via `atLeastOneIsNaN`.)*
- [ ] **Deliberate divergence, ADR-014 discipline — finite operands only**: arithmetic evaluates in native `Double`. Catroid computes PLUS/MINUS/MULT/DIVIDE via `BigDecimal(…, MathContext.DECIMAL128)` (`FormulaElement.java`), so e.g. Catroid's `0.1 + 0.2` is exactly decimal 0.3 while binary `Double` gives 0.30000000000000004. At embroidery-relevant magnitudes this rounding difference is far below resolution; it grows with magnitude, so the ADR pinned during this story states the accepted scope precisely, and formula tests assert with ADR-014-style tolerance where binary/decimal representations differ. The stitch-visible **non-finite normalization is mirrored, not diverged from** (previous criterion).
- [ ] `interpretInteger(scope:)` = `interpretDouble` truncated **toward zero** for in-range values (Java `Double.intValue()`), and **saturating to Java's `Int32` bounds** (−2 147 483 648 … 2 147 483 647) for out-of-range magnitudes and ±∞ — `Double.intValue()` is a narrowing int conversion that saturates at those bounds where Swift's `Int(_:)` traps (a platform difference the engine already documents in `StitchPattern.swift`'s `maxStitchesPerUpdate` comment). This is the contract for running/triple stitch length (Catroid `RunningStitchAction`/`TripleStitchAction` use `interpretInteger`). `interpretFloat(scope:)` returns `Float` — zigzag length **and** width use it (`ZigZagStitchAction` uses `interpretFloat` for both).
- [ ] Variable scoping: object-scoped variables shadow same-named project-scoped ones (Catroid `UserDataWrapper.getUserVariable`: sprite-first, then project). Unknown variable evaluates to 0. The evaluator takes a read-only `Scope`; mutation belongs to the interpreter (US-205).

## Test-first plan
1. Literal and each operator on known inputs, including `pow`; unary minus. Where binary `Double` and Catroid's decimal128 differ (e.g. `0.1 + 0.2`), assert with the ADR-014 tolerance per the divergence AC.
2. Structural nesting evaluates in tree order: `binary(.mult, number(2), binary(.plus, number(3), number(4)))` = 14 (precedence lives in structure, not re-parsing).
3. `interpretInteger` truncates toward zero: 2.9 → 2, −2.9 → −2 (not floor); out-of-range: 1e19 and `+∞` saturate to `Int32.max` (2 147 483 647), −1e19 and `−∞` to `Int32.min` — no trap.
4. Any division by a zero divisor (`1/0`, `0/0`) throws — Catroid's DIVIDE explicitly returns NaN for a zero divisor and `assertNotNaN` rejects it; NaN is sticky (a `0/0` subtree inside a PLUS still throws, as does `unaryMinus(0/0)`). A bare `pow` overflow (e.g. `1e308^2`) evaluates to `Double.greatestFiniteMagnitude` **without** throwing, and `pow(1e308, 2) + 1` also evaluates to `Double.greatestFiniteMagnitude` — per-node normalization converts the ∞ before it becomes an operand (amended 2026-07-19, see AC above). The error type is caught-able by brick execution (US-205's fallback-and-continue relies on it).
5. Variable resolution: project variable read; object variable shadows a same-named project variable; unknown variable → 0.
